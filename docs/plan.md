# FreeGrow 프론트엔드 지원용 사이드 프로젝트 계획

## Context

**왜 이 계획이 필요한가**
지원자(3년차 해양 GIS SI 개발자, 백엔드 위주 — Spring Boot/PostgreSQL/JSP/React Native, React는 독학 수준)가 (주)프리그로우(서비스명 GrowSpace)의 **프론트엔드 개발자** 직군에 지원하면서 포트폴리오에 첨부할 사이드 프로젝트를 설계한다.

**공고 핵심 (사람인 rec_idx=53424352)**
- 회사: FreeGrow / GrowSpace — UWB 기반 실내외 위치추적, 스마트 물류·산업 안전·공간 데이터 플랫폼 (부산, 딥테크 스타트업)
- 필수: Flutter **또는** React 실무 3년+ (둘 다면 1~2년+) / REST / 상태관리 / Git
- 우대: 실시간 데이터 시각화(지도·대시보드), WebSocket·MQTT, BLE·UWB, IoT 센서 연동, RTLS, 하이브리드 앱(Flutter Web/PWA), AI 툴 활용, UI/UX 주도성
- 가산점 포트폴리오: ① **Flutter+React 둘 다 있는 프로젝트**, ② **실시간 데이터 시각화**, ③ **지도 기반 UI**, ④ 직접 기획부터 한 사이드 프로젝트

**전략적 진단**
- 지원자에게 React 실무 경험이 약하고 Flutter 경험은 없음 → 격차 존재.
- 그러나 **해양 GIS 3년 경력**은 회사가 강조하는 "공간 데이터·실시간 위치·지도 기반 UI"와 정확히 같은 영역이라 **비대칭 강점**으로 전환 가능.
- 따라서 단일 프로젝트에 React 웹 + Flutter 모바일 + 실시간 위치 시각화 + 지도 UI를 모두 담아 공고와 1:1 적중시키는 "정면돌파" 방향을 채택.

**최종 의도**
"해양 GIS의 공간 데이터·GPS 노이즈 처리 경험을 실내 RTLS 도메인으로 확장 시도"라는 서사로, 회사 제품(GrowSpace)의 미니어처 시스템을 직접 만들어 면접에서 1:1 매칭으로 어필한다.

## 프로젝트 개요

**프로젝트명 (작업명)**: GrowSpace 미니어처 — UWB RTLS 시뮬레이터 & 실시간 시각화 대시보드

**시나리오**: **창고/공장 실내 자산 추적 (RTLS 시뮬레이션)** — 회사가 실제 푸는 문제와 동일한 시나리오를 직접 구현. 흉내가 아닌 진짜로 만들기 위해 **데이터 충실도 우선** 전략 채택 (노이즈·드리프트·신호 손실을 시뮬레이터에서 재현, 클라이언트에서 칼만 필터/이동평균으로 보정).

**왜 ②(실내 RTLS)인가**: 회사 도메인 정면 적중. "딥테크 흉내" 위험은 (C) 데이터 충실도 우선으로 상쇄.

**왜 (C) 데이터 충실도 우선인가**: GIS에서 GPS 노이즈를 다뤄본 경험과 직결되어 면접 답변 깊이가 가장 깊고, 우대사항(실시간 데이터 시각화·무선 통신 프로토콜 이해)을 동시에 적중.

## 시스템 아키텍처

### Phase 1 (React 웹 대시보드만)

```
[Express 서버]
  ├─ 내부 시뮬레이터 모듈 (Node TS, 노이즈/드리프트/신호 손실 재현)
  │     ↓ in-process tick (setInterval)
  └─ WebSocket 게이트웨이
        ↓ ws push
  [React 웹 대시보드] (관리자)
        ⤷ 클라이언트측 칼만/이동평균 보정
        ⤷ react-konva 평면도 + Leaflet 옥외 보조 화면
```

### Phase 2 (Flutter 앱 합류)

```
[Flutter 앱]                  [Express 서버]
  GPS 위치 발행 ----MQTT publish---> [Mosquitto 브로커]
                                              ↓ MQTT subscribe
                                       [Express 서버]
                                       ├─ 내부 시뮬레이터 (옥내 자산용)
                                       └─ WebSocket 송신
                                              ↓
                                  [React 웹 대시보드]
                                  [Flutter 앱]도 WebSocket 구독 (자기·동료 위치 표시)
```

**구성 결정 근거**
- **Phase 1: 시뮬레이터 = 백엔드 내부 모듈**. 별도 프로세스·MQTT 브로커 불필요. 실행 환경 단순화로 4주 일정 안정화.
- **Phase 2: MQTT는 Flutter 앱이 GPS를 백엔드로 발행하는 채널**로 도입. 흉내가 아니라 **실제 모바일 디바이스 → 백엔드** 진짜 IoT 통신이 됨 → 우대사항(MQTT, 하드웨어/IoT 연동, RTLS) 정직 적중도 상승.
- **WebSocket(서버 → 프론트)은 Phase 1부터 유지** — 실시간 푸시 통신 어필은 Phase 1 시점에 이미 충족.
- **백엔드 Express** (Node 단일 런타임, 시뮬레이터·서버·WebSocket·MQTT 구독을 한 코드베이스에서 관리).
- **위치 보정은 클라이언트(React)에서** → "백엔드는 원시 데이터를 흘리고 클라이언트가 노이즈를 흡수해 부드러운 궤적을 그린다"는 스토리. 프론트엔드 책임 영역에 대한 깊은 이해 어필.

### Phase 1 시점 MQTT 어필 보강 전략
- README에 **Phase 2 아키텍처 다이어그램** (Flutter→MQTT→백엔드)을 함께 명시.
- 가능하면 Phase 2 착수 전 Flutter 1화면 스파이크에서 **MQTT 발행 코드만이라도 시연** 영상으로 첨부.

## 기술 스택 (Phase 1: React 웹 대시보드)

### 백엔드 (시뮬레이터 통합형)
- **Express + TypeScript**
- **시뮬레이터는 백엔드 내부 모듈로 구현** (별도 프로세스·MQTT 미사용)
  - 노이즈 모델: 가우시안 노이즈 + 드리프트 + 간헐적 신호 손실 재현
  - `setInterval` 기반 tick으로 위치 데이터 생성 후 WebSocket으로 직접 푸시
  - 모듈 인터페이스를 **데이터 소스 추상화**로 설계 (Phase 2에서 MQTT 구독자가 같은 인터페이스로 들어올 수 있게)
- WebSocket 송신 (`ws` 패키지)
- 인증 없음 (간소화)
- REST API: 앵커 목록 / 작업자 목록 / 과거 궤적 조회 (메모리 또는 SQLite, 결정 보류)
- **Phase 2 추가**: `mqtt` 패키지로 Mosquitto 브로커 구독, Flutter가 발행한 GPS를 받아 동일 WebSocket 파이프라인에 합류

### 프론트엔드 (React 웹)

| 영역 | 선택 |
|---|---|
| 언어 | TypeScript |
| 서버 상태 | **React Query** (REST 캐싱·재시도) |
| 클라이언트 상태 | **Zustand** (UI 상태 + WebSocket 실시간 위치 맵) |
| HTTP | **fetch** (인증 없음 + 의존성 최소화) |
| 실내 시각화 | **react-konva** (평면도 위 마커·궤적, 60fps 100+ 객체 대응) |
| 옥외 보조 화면 | **Leaflet** (지도 기반 UI 우대 적중 + GIS 경력 직결) |
| 라우팅 | React Router |
| 스타일 | (보류 — Phase 1 착수 전 결정. 후보: Tailwind / CSS Modules) |

- **Phase 1**: 로컬에서 백엔드(`npm run dev`) + 프론트(`npm run dev`) 두 프로세스만 실행. 시뮬레이터·MQTT 브로커 의존성 없음 → **docker-compose 없이도 충분**.

## Phase 2: Flutter 앱 (Phase 1 완료 후 착수)

**역할**: 작업자 시점 + **GPS 데이터 발행자(퍼블리셔)**
- 웹 = 관리자 모니터링 / 모바일 = 작업자
- Flutter 앱이 디바이스 GPS를 읽어 **MQTT로 백엔드에 발행** → 백엔드가 구독 후 WebSocket으로 모든 클라이언트에 브로드캐스트
- 즉 모바일 앱은 단순 시청자가 아니라 **실제 IoT 디바이스 역할**을 수행 → 우대사항(MQTT, 하드웨어/IoT 연동, BLE/UWB 토폴로지 이해, RTLS) 정직 적중

**범위 (3~4화면)**:
1. 본인 위치(GPS) 발행 + 주변 동료 위치를 평면도/지도에 표시
2. 위험구역 진입 시 푸시/진동 알림
3. 간단한 작업 체크인/체크아웃

**기술**:
- Flutter + `mqtt_client` 패키지로 MQTT 발행
- 디바이스 GPS는 `geolocator` 패키지
- WebSocket 구독으로 동료 위치 수신
- 백엔드 측은 `mqtt` 패키지로 Mosquitto 구독 추가 (Phase 1의 데이터 소스 추상화 인터페이스에 합류)
- Mosquitto는 docker-compose에 추가

**기간**: ~2주

**주의**: Phase 1 지원 시점에 Flutter가 *완전히 비어 있으면* 가산점 1순위(Flutter+React 둘 다) 및 MQTT 우대사항을 잃음. 따라서 **이력서/포트폴리오에는 "Phase 2: Flutter 앱 + MQTT GPS 발행 (진행 중)" 명시 + 가능하면 Flutter 1화면 스파이크에서 MQTT 발행 코드만이라도 시연**.

## 검증 기준

1. **Phase 1 MVP (Week 2 종료)**: 백엔드(`npm run dev`) + 프론트(`npm run dev`) 두 프로세스 실행 후 브라우저에서 평면도 위 점이 실시간으로 움직이는 것 확인.
